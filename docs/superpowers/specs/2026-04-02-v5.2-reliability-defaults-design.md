# v5.2 Reliability & Defaults — Design Spec

> Milestone: 5 фаз, 2 бага + 5 фич из анализа OpenClaw v2026.4.1.

---

## Контекст

Анализ OpenClaw v2026.4.1 выявил два реальных бага в HydeClaw и 5 фич с высоким ROI. Milestone v5.2 закрывает всё в одном цикле.

**Баги:**
- Anthropic thinking blocks полностью теряются при десериализации — нет варианта в enum, нет поля в БД
- Memory reindex охватывает только хардкодированный `zettelkasten/`, игнорирует весь остальной workspace и session transcripts

**Фичи из OpenClaw:**
- `agents.defaults.params` — глобальные LLM параметры по умолчанию
- `compaction` capability в Active Providers — отдельная модель для компакции контекста
- Per-cron tool allowlist — ограничение инструментов на уровне job
- Telegram error policy — экспоненциальный backoff и per-chat cooldown

## Phase 32: Anthropic Thinking Blocks

**Цель:** Thinking blocks сохраняются в БД, восстанавливаются при replay и корректно обрабатываются при компакции.

**Проблема:** `AnthropicContentBlock` enum не имеет `Thinking` варианта — все thinking blocks из API попадают в `Other` и молча отбрасываются. `MessageRow` не имеет поля для хранения.

**Изменения:**

1. **`providers_anthropic.rs:178`** — добавить вариант:
   ```rust
   #[serde(rename = "thinking")]
   Thinking { thinking: String, signature: String },
   ```
   В `parse_anthropic_response` — собирать thinking blocks отдельно, возвращать вместе с text/tool_calls.

2. **Миграция** — новая колонка в `messages`:
   ```sql
   ALTER TABLE messages ADD COLUMN thinking_blocks JSONB DEFAULT NULL;
   ```
   Формат: `[{"thinking": "...", "signature": "..."}]`

3. **`db/sessions.rs`** — `MessageRow` + `save_message()` + `load_messages()` — сохранять и восстанавливать `thinking_blocks`.

4. **`engine.rs` build_context для Anthropic** — при построении контекста вставлять thinking blocks обратно в content array перед `tool_use` блоками (требование Anthropic API).

5. **`history.rs` компакция** — thinking blocks не включаются в `format_messages_for_compaction` (они уже зафиксированы как выводы в text); при compaction через non-Anthropic модель — dropping is correct, signature не переносится.

**Success criteria:**
1. Агент с `thinking: enabled` корректно работает в многоходовых сессиях без ошибок `invalid_request_error`
2. После reload страницы thinking blocks в истории не ломают контекст
3. `cargo test` проходит

## Phase 33: Memory Reindex — Universal Sources

**Цель:** Reindex охватывает весь workspace (кроме системных директорий) и session transcripts.

**Проблема:** Хардкод `"zettelkasten"` в `reindex.rs`, `memory.rs` (watch), `engine_memory.rs`. Session transcripts никогда не индексируются. `clear_existing=true` стирает без восстановления.

**Системные директории (исключить из индексации):**
```rust
const MEMORY_INDEX_EXCLUDE_DIRS: &[&str] = &["tools", "skills", "mcp", "uploads", "agents"];
```
Константа добавляется в `workspace.rs` рядом с `SHARED_ROOT_DIRS`.

**Изменения:**

1. **`reindex.rs`** — заменить хардкод: рекурсивный обход всего workspace, пропуская `MEMORY_INDEX_EXCLUDE_DIRS`. Индексировать только `.md` и `.txt` файлы.

2. **`memory.rs` (watch)** — watch на весь workspace root, фильтровать события из системных директорий через `MEMORY_INDEX_EXCLUDE_DIRS`.

3. **`engine_memory.rs`** — убрать передачу `directory: "zettelkasten"` в параметры задачи. Добавить `include_sessions: bool` (default `true`).

4. **`reindex.rs` sessions** — если `include_sessions: true`: выгрузить session transcripts из БД за последние 90 дней (configurable через `[memory] session_index_days = 90`), форматировать как `[role]: content`, индексировать с source `session:{session_id}`.

5. **Обратная совместимость** — старые задачи с `"directory"` в JSONB params продолжают работать (fallback на legacy поведение если поле присутствует).

**Success criteria:**
1. Reindex без параметров индексирует все `.md`/`.txt` файлы в workspace кроме системных директорий
2. Session transcripts попадают в `memory_chunks` при reindex
3. Watch триггерится на изменения в любой не-системной директории workspace
4. `cargo test` проходит

## Phase 34: Global Agent Defaults + Compaction Provider

**Цель:** Глобальные LLM параметры по умолчанию и отдельный провайдер для компакции контекста.

### Часть A: Global Agent Defaults

**`hydeclaw.toml`:**
```toml
[agent.defaults]
temperature = 0.7
max_tokens = 4096
# любые другие параметры поддерживаемые провайдером
```

**Приоритет:** routing rule params → agent config params → `[agent.defaults]` → provider defaults.

**Изменения:**
- `config/mod.rs` — новая структура `AgentDefaults { temperature: Option<f64>, max_tokens: Option<u32> }`, поле в `Config`
- `providers.rs` — fallback логика при создании провайдера

### Часть B: Compaction Provider

**Архитектура:** новая capability `compaction` в таблице `provider_active`.

**Изменения:**

1. **Миграция** — добавить `'compaction'` в constraint допустимых capabilities в `provider_active`.

2. **UI** — в секции ACTIVE PROVIDERS добавить строку `Compaction` с дропдауном (аналогично Embedding). Если не выбран — показывать `(агентский провайдер)`.

3. **`history.rs`** — функция `compact_if_needed()` принимает `compaction_provider: Option<Arc<dyn LlmProvider>>`. Если `Some` — использует его для summarization и fact extraction. Если `None` — поведение как сейчас.

4. **`engine.rs`** — при инициализации читать активного провайдера для capability `compaction` из `provider_active`, передавать в `compact_if_needed()`.

**Success criteria:**
1. `[agent.defaults]` применяется когда agent config не задаёт temperature/max_tokens
2. Compaction через haiku/qwen при активном Compaction провайдере, без изменения основного агентского провайдера
3. Если Compaction провайдер не задан — поведение идентично текущему

## Phase 35: Per-Cron Tool Allowlist

**Цель:** Каждый scheduled job может ограничивать набор инструментов агента.

**Изменения:**

1. **Миграция:**
   ```sql
   ALTER TABLE scheduled_jobs ADD COLUMN tool_policy JSONB DEFAULT NULL;
   ```
   Формат: `{"allow": ["memory_search", "searxng_search"], "deny": []}` — те же поля что у `AgentToolPolicy`.

2. **`config/mod.rs`** — `ScheduledJob` добавить `pub tool_policy: Option<AgentToolPolicy>`.

3. **API** (`POST/PUT /api/cron`) — принимать `tool_policy` в теле запроса, сохранять в БД.

4. **`scheduler/mod.rs`** — при запуске job передавать `tool_policy` override в engine. Engine применяет job policy поверх агентской (job policy более restrictive: deny объединяется, allow пересекается).

5. **UI** — в диалоге создания/редактирования cron job добавить collapsible секцию "Tool Policy" с двумя textarea (Allow / Deny), аналогично настройкам агента.

**Success criteria:**
1. Job с `allow: ["memory_search"]` использует только `memory_search`, остальные инструменты недоступны
2. Job без `tool_policy` — поведение идентично текущему
3. API валидирует имена инструментов (паттерн `[a-zA-Z0-9_-]`)

## Phase 36: Telegram Error Policy

**Цель:** Повторяющиеся ошибки доставки подавляются per-chat/topic, разные ошибки не блокируются, 429 уважает `retry_after`.

**Изменения:**

1. **`channels/src/drivers/telegram.ts` — `retryTg()`:** заменить линейный backoff (`500ms × i`) на экспоненциальный (`1000ms × 2^i`, cap 30s).

2. **Per-chat cooldown map:**
   ```typescript
   type CooldownEntry = { errorCode: number; cooldownUntil: number };
   const chatCooldowns = new Map<string, CooldownEntry>(); // key: `${chatId}:${threadId??''}`
   ```
   Логика: если та же `errorCode` в том же чате — пропустить до `cooldownUntil`. Разная `errorCode` — не блокировать.

3. **429 handling** — читать `retry_after` из тела ответа Telegram (`error.parameters?.retry_after`), использовать как cooldown если присутствует. Иначе — `error_cooldown_ms` из конфига.

4. **`channels.toml` секция `[telegram]`:**
   ```toml
   [telegram]
   error_cooldown_ms = 60000
   error_policy = "suppress_repeated"   # "suppress_repeated" | "always_retry"
   ```
   Defaults применяются если секция отсутствует.

**Success criteria:**
1. При повторной отправке в забаненный чат (403) — сообщения пропускаются на `error_cooldown_ms`, не накапливаются в очереди
2. 429 с `retry_after: 30` → cooldown ровно 30s
3. Разные ошибки (403 в чате A, 500 в чате B) не блокируют друг друга
4. `bun test` проходит
