    **Content hash utility** (in chat-store.ts, near the utility functions section around line 130):

    1. Add a `contentHash` function that creates a lightweight fingerprint of a ChatMessage array:
       ```typescript
       export function contentHash(messages: ChatMessage[]): string {
         // Hash based on id + role + text content of parts (ignoring timestamps)
         let hash = 0;
         for (const m of messages) {
           const str = m.id + m.role + m.parts.map(p => p.type === "text" ? p.text : p.type + (("toolCallId" in p) ? p.toolCallId : "")).join("|");
           for (let i = 0; i < str.length; i++) {
             hash = ((hash << 5) - hash + str.charCodeAt(i)) | 0;
           }
         }
         return hash.toString(36);
       }
       ```
       This is a fast djb2-style hash. No crypto needed — we're comparing live vs history for render optimization, not security.

    2. Add a `reconcileLiveWithHistory` function:
       ```typescript
       export function reconcileLiveWithHistory(
         live: ChatMessage[],
         history: ChatMessage[]
       ): ChatMessage[] | null {
         // If IDs and content match, return null (skip update)
         if (live.length === history.length && contentHash(live) === contentHash(history)) {
           return null; // identical — skip re-render
         }
         // History has different content or more messages — use history
         return history;
       }
       ```

    - After React Query has had time to populate cache (~500ms), transition to history mode

              }

          }

      }
    }

    **OPTI-04: User message ID stabilization** (chat-store.ts, in `data-session-id` handler around L755):

    4. The `data-session-id` SSE handler already confirms the user message (sets `status: "confirmed"`). Per OPTI-04, the user message keeps its client UUID. The history load uses the DB-assigned ID. Since assistant messages already get their DB ID from `event.messageId` in the `start` event, the only mismatch is user messages. This is acceptable — per CONTEXT.md decision: "user message gets client UUID but is matched by content+role during history load." No change needed here; the existing behavior is correct.

    However, to improve DOM reuse: when `messageSource` transitions from "live" to "history" and history is loaded via convertHistory, user messages get DB IDs. For React key stability, we should ensure the live-to-history transition does not cause unnecessary DOM unmount/remount of user message elements. Since we're using a delayed transition (step 3 above) and content hash comparison, and since Virtuoso uses `computeItemKey={(index, item) => item.id}`, the user message DOM nodes WILL be unmounted and remounted with new keys (DB IDs vs client UUIDs). This is acceptable per CONTEXT.md: "No transition effect — if IDs match, React reuses DOM nodes." For user messages, IDs won't match, but since the content is identical, the visual result is the same (no flicker, just a DOM swap).

    To fully implement OPTI-04 for user messages, we could update the `data-session-id` handler to fetch the user message's DB ID from the first history response. But this requires an additional API call or convention. Since CONTEXT.md says this is acceptable behavior ("matched by content+role during history load"), we document this as the expected behavior.
