    1. `resumeStream` sets messageSource to `{ mode: "live", messages: [] }` with NO history seed — so the UI shows an empty list + thinking indicator (ghost avatar)
    2. `useSessionMessages` is blocked while `connectionPhase` is active — so history never loads during the resume window

    **Fix A: resumeStream seeds history from React Query cache before connecting**

    In `resumeStream()`, before setting connectionPhase to "streaming":

    ```typescript
    function resumeStream(agent: string, sessionId: string) {
      const st = get().agents[agent];
      if (st && isActiveStream(st.streamStatus)) return;

      abortActiveStream(agent);
      streamGenerations[agent] = (streamGenerations[agent] ?? 0) + 1;
      const myGeneration = streamGenerations[agent];
      const controller = new AbortController();
      agentAbortControllers[agent] = controller;

      // Seed with history from React Query cache so UI shows messages immediately
      const cachedHistory = getCachedHistoryMessages(sessionId);
      const seedMessages = cachedHistory.length > 0 ? cachedHistory : [];

      update(agent, {
        messageSource: { mode: "live", messages: seedMessages },
        streamStatus: "streaming",
        streamError: null,
        connectionPhase: "streaming",
        connectionError: null,
      });

      // Remove sessionStorage persistence flag (Phase 41 already cleans this on finish,
      // but also clean here to prevent stale keys from previous crashes)
      sessionStorage.removeItem(`hydeclaw.streaming.${agent}`);

      // ... rest of fetch logic unchanged
    }
    ```

    This way, pressing F5 immediately shows the cached conversation + thinking indicator (correct behavior).

    **Fix B: On 204 (engine done), transition to history mode**

    When resumeStream gets a 204 response (engine already finished), instead of just setting idle, set messageSource back to history so useSessionMessages takes over with fresh data:

    ```typescript
    if (resp.status === 204) {
      update(agent, {
        streamStatus: "idle",
        connectionPhase: "idle",
        messageSource: { mode: "history", sessionId },
      });
      return;
    }
    ```

    This ensures that after resume discovers the engine is done, the UI fetches fresh history from the server (which may have newer data than the stale cache).

    **Fix C: Fix sourceMessages priority in ChatThread.tsx**

    The current logic:
    ```typescript
    const sourceMessages = liveMessages.length > 0 ? liveMessages : historyMessages;
    ```

    After plan 01, `liveMessages` reads from `messageSource`. The logic should be:
    ```typescript
    const sourceMessages = useMemo(() => {
      if (messageSource.mode === "live" && messageSource.messages.length > 0) {
        return messageSource.messages;
      }
      if (messageSource.mode === "history") {
        return historyMessages;
      }
      // new-chat or empty live
      return historyMessages.length > 0 ? historyMessages : [];
    }, [messageSource, historyMessages]);
    ```

    This eliminates the race where both sources compete. The messageSource.mode is the single authority.

    **Fix D: Prevent historyLoading skeleton when cache is warm**

    The current `if (historyLoading)` guard shows a skeleton. But on F5 with a cached session, React Query has stale data immediately (staleTime). The skeleton should only show when there is truly no data:

    ```typescript
    if (historyLoading && !sessionMessagesData && messageSource.mode !== "live") {
    ```

    This prevents the skeleton from flashing when cached data exists or when we're in live mode with seeded messages.

    **Fix E: Clean up sessionStorage flag in resumeStream**

    Remove the `sessionStorage.setItem` line from `resumeStream` — Phase 41 already eliminated the sessionStorage-based thinking indicator. If the line still exists, remove it. The `connectionPhase` is the single source of truth for thinking state.

    **Fix F: Ensure selectSession clears stale live messages**

    In `selectSession`, when switching to a different session, the messageSource should be set to `{ mode: "history", sessionId }`. Verify this was done correctly in plan 01 and that no stale `messages` array leaks through.

    **IMPORTANT anti-pattern to avoid (Pitfall 8):** Never call `get()` inside `set(draft => ...)`. Read `getCachedHistoryMessages` result before the `update()` call.
